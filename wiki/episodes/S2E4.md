# S2E4: Memory Management for AI Agents

## Summary
This episode tackles one of the core engineering challenges in building AI agents: the context window is finite, and even large ones have structural biases that make the middle hard to use. The episode builds on S2E3's "lost in the middle" finding to ask: given this constraint, how do production systems actually manage what an agent knows and remembers over long, complex tasks?

The episode traces a conceptual arc from the MemGPT paper (UC Berkeley, 2023) — which borrowed the operating system concept of virtual memory to give agents active, self-managed memory — through to the more pragmatic compaction mechanism used in tools like Claude Code today. Along the way, it distinguishes MemGPT-style active memory management from the better-known RAG (retrieval augmented generation) pattern, and introduces the idea of a context hierarchy: not all content in a context window has equal standing, and skilled system designers exploit this deliberately.

The practical upshot is the concept of "context engineering" as a discipline — understanding that the context window is a resource to be managed, not a flat space to fill. Knowing how compaction works, where it fails (context rot), and how to use mechanisms like CLAUDE.md to pin durable information are presented as genuinely useful skills for anyone working with agent systems today.

## Key Concepts
- **Context Window**: The fixed amount of text/tokens an LLM can "see" at once, acting as the agent's working memory → see also: concepts/context-window.md
- **MemGPT**: A UC Berkeley architecture that applies OS virtual memory principles to LLM agents, letting the agent actively manage what's in context vs. external storage → see also: concepts/memgpt.md
- **Virtual Memory / Paging**: The OS technique of moving data between fast RAM and slower disk to create the illusion of larger memory; used as the conceptual model for MemGPT → see also: concepts/virtual-memory-paging.md
- **RAG (Retrieval Augmented Generation)**: A pattern where an external system retrieves relevant information and injects it into the model's context before generation; contrasted with MemGPT as retrieval *done to* the model rather than *by* it → see also: concepts/retrieval-augmented-generation.md
- **Compaction**: The mechanism by which an agent summarizes and replaces its conversation history when approaching context limits, enabling long sessions to continue → see also: concepts/context-compaction.md
- **Context Rot**: A failure mode where repeated compaction (summaries of summaries) causes accumulated detail loss, leading to fuzzy or contradictory agent behavior → see also: concepts/context-rot.md
- **Context Hierarchy**: The structured layering of content in an agent's context window — system prompt, pinned project instructions (e.g. CLAUDE.md), then conversation history — where position and durability confer different levels of attention → see also: concepts/context-hierarchy.md
- **Memory Taxonomy (Semantic / Episodic / Procedural)**: A framework for distinguishing what kind of information different memory systems store: RAG handles factual world knowledge, MemGPT handles task-event memory, compaction compresses episodic into semantic, and skills handle procedural know-how → see also: concepts/agent-memory-taxonomy.md

## Takeaways
1. **MemGPT reframes memory management as an OS problem**: The context window is RAM; external storage is disk; the agent itself is the operating system deciding what to page in and out — using the same tool-calling architecture it uses for everything else.
2. **RAG and MemGPT solve different problems**: RAG is retrieval *done to* the model by an external system; MemGPT is retrieval *done by* the model. The agent in MemGPT is aware of its own memory architecture and actively manages it.
3. **Compaction is useful but lossy**: The "infinite context" framing is partly marketing. Compaction creates a rolling window of summarized history. What survives are high-level decisions and task state; what gets lost are specific reasoning chains, subtle constraints, and exact exceptions agreed upon earlier.
4. **CLAUDE.md is pinned memory**: By storing project conventions and architectural constraints in a file that gets re-injected at every compaction, users can guarantee certain context survives — turning an understanding of compaction mechanics into a practical power-user technique.
5. **The context hierarchy isn't flat**: System prompts, pinned instructions, and conversation history occupy different positions and carry different weight. Skilled agent designers put the most important information where the architecture is already paying attention — at the beginning of the context.
